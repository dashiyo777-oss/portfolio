## 1. システム概要

統覧 TORAN は、日本の国会議員・政治家 816名を8つの評価軸で定量化し、100点満点のスコアと根拠データとともに公開する評価サイトです。

### 基本方針

| 原則 | 内容 |
|------|------|
| **客観性** | すべてのスコア変更は根拠（ソースURL付き）が必要 |
| **透明性** | 評価根拠（Evidence）を公開し、誰でも確認可能にする |
| **更新性** | 選挙・不祥事・実績に応じて随時見直す |
| **一貫性** | 全議員に同じ基準を適用する |

### 評価の対象

- 衆議院議員・参議院議員（現職・元職）
- 国政に影響のある地方政治家（一部）

## 2. 評価の仕組み（8軸評価）

議員を以下の8つの軸でそれぞれ **0〜5点** で評価します。

| 軸番号 | 軸名 | 評価の観点 |
|--------|------|-----------|
| [0] | **国民生活改善** | 生活水準の向上、福祉・医療・教育の充実への貢献 |
| [1] | **経済・財政** | GDP成長への貢献、財政健全化、税・予算政策の実績 |
| [2] | **安全保障** | 防衛政策、外交実績、経済安保への取り組み |
| [3] | **政策実現力** | 立法・行政での政策遂行能力、大臣・要職での実績 |
| [4] | **説明責任** | 情報公開の姿勢、透明性、国民への説明 |
| [5] | **長期国益** | 少子化対策、食料安保、エネルギー安保、次世代への投資 |
| [6] | **公共性** | 利益相反の回避、政党を超えた公益優先の姿勢 |
| [7] | **政治倫理** | 法令遵守、廉潔性、政治資金の適切な管理 |

### 各軸のスコア基準

| スコア | 基準 |
|--------|------|
| **5** | 顕著な実績あり。客観的に証明できる高い成果。 |
| **4** | 明確な実績あり。複数の根拠で裏付けられる。 |
| **3** | 標準的。特筆すべき実績も問題もない。 |
| **2** | やや不十分。実績が乏しいか課題がある。 |
| **1** | 問題あり。具体的な課題・懸念が確認されている。 |
| **0** | 深刻な問題あり（逮捕・有罪・重大な倫理違反等）。 |

> **重要**: スコアを4以上にする場合、または2以下にする場合は必ず具体的な根拠（Evidence）が必要です。

## 3. スコア計算とランク判定

### 総合スコア計算式

```
総合スコア = 8軸の合計 × 100 ÷ 40（小数点以下切り捨て）
```

例: axes = [4, 5, 3, 4, 3, 4, 4, 3] → 合計30 → 30×100÷40 = **75点**

**注意**: 整数値の制約により、すべての点数が実現可能ではありません。

| 8軸合計 | 総合スコア |
|--------|----------|
| 40 | 100点 |
| 36 | 90点 |
| 30 | 75点 |
| 28 | 70点 |
| 24 | 60点 |
| 20 | 50点 |
| 16 | 40点 |

### ランク判定表

| ランク | 総合スコア | 目安 |
|--------|-----------|------|
| **S** | 90点以上 | 国内トップクラスの実績 |
| **A+** | 87〜89点 | 非常に優秀 |
| **A** | 83〜86点 | 優秀 |
| **A-** | 80〜82点 | 良好（上位） |
| **B+** | 77〜79点 | 標準以上 |
| **B** | 73〜76点 | 標準（上位） |
| **B-** | 70〜72点 | 標準 |
| **C+** | 67〜69点 | やや低め |
| **C** | 63〜66点 | 低め |
| **C-** | 60〜62点 | 要改善 |
| **D** | 59点以下 | 実績・情報が乏しい |
| **情報不足** | — | 評価に必要な情報が足りない |

## 4. 評価の種類と品質基準

### 評価ステータス一覧

| ステータス | 意味 | 品質レベル |
|-----------|------|-----------|
| **評価済** | フルリサーチ済み。根拠付きの詳細評価。 | ★★★ 高 |
| **Wikipedia基礎評価** | Wikipediaの公開情報から自動・半自動で生成した基礎評価。 | ★★ 中 |
| **情報不足** | 公開情報が少なく、正確な評価が困難。 | ★ 低 |

### Wikipedia基礎評価の自動スコアリングロジック

Wikipedia基礎評価は以下のロジックで自動生成されます（全軸デフォルト2点から開始）:

**加点条件**
- 当選2〜3回: 国民生活改善 +1
- 当選4回以上: 国民生活改善 +1、説明責任 +1
- 大臣・長官就任: 政策実現力 +1、長期国益 +1
  - 経済財政担当: 経済・財政 +1
  - 防衛・外交担当: 安全保障 +1
  - 厚労・福祉担当: 国民生活改善 +1
  - 少子化・地方担当: 長期国益 +1
- 副大臣・政務官: 政策実現力 +1
- 委員会委員長・党幹部: 公共性 +1

**制約**: Wikipedia基礎評価では各軸の上限は3点（それ以上は追加根拠が必要）

## 5. フラグシステム（犯罪・要注意）

### 犯罪フラグ（🚨 flag_crime）

**発動条件**: Wikipediaまたは根拠データに以下のキーワードが含まれる場合  
`逮捕` `起訴` `有罪` `刑事事件`

**スコアへの影響**:
- 政治倫理軸[7]: -2（最低0）
- 説明責任軸[4]: -1（最低0）

**表示**: サイト上に🚨マークで警告表示

### 要注意フラグ（⚠️ flag_caution）

**発動条件**: Wikipediaまたは根拠データに以下が含まれる場合  
`政治資金問題` `不記載` `裏金` `統一教会接点` `収賄` `選挙違反`  
`政治資金規正法違反` `離党勧告` `除名処分`

**スコアへの影響**:
- 政治倫理軸[7]: -1（最低1）

**表示**: サイト上に⚠️マークで注意表示

> **注意**: フラグは一度立てたら削除しない。情報が誤りであることが確認された場合のみ見直す。

## 6. 根拠データ（Evidence）の書き方

評価の根拠となる情報を以下の形式で記録します。

### Evidenceフィールド一覧

| フィールド | 必須 | 説明 | 例 |
|-----------|------|------|----|
| **cat** | ○ | カテゴリ | "実績" / "問題・疑惑" / "発言・主張" |
| **sub** | ○ | サブカテゴリ | "政策実現" / "犯罪・違反" / "政治資金問題" |
| **summary** | ○ | 一行要約（100字以内） | "財務大臣として28年ぶりPB黒字化を達成" |
| **detail** | ○ | 詳細説明（500字程度） | 具体的な経緯・数字・背景 |
| **src** | ○ | ソース名 | "Wikipedia" / "NHK" / "財務省資料" |
| **url** | △ | 参照URL | "https://ja.wikipedia.org/wiki/○○" |
| **rel** | ○ | 関連する軸 | "政策実現力" / "政治倫理" |
| **impact** | ○ | 重要度 | "高" / "中" / "低" |
| **date** | ○ | 実績・報道日付 | "2025-10" / "2026-01-15" |

### Evidenceの質について

**良いEvidence（推奨）**
- 具体的な数字・年月がある（「28年ぶりPB黒字化」「122兆円」）
- 安定したURLがある（Wikipedia・官公庁・大手報道）
- 客観的事実を記載（賛否が分かれる解釈を含まない）

**悪いEvidence（避けるべき）**
- 「優れた政治家」「有能な議員」など主観的表現
- ソースURLのないもの（Google検索URLは不可）
- 「〜と言われている」などの伝聞
- 同一人物を別のサイトが転載しただけのもの

### カテゴリ・サブカテゴリ一覧

**実績**
- 政策実現 / 経済財政 / 安全保障 / 地方創生 / 福祉・医療 / 教育 / 環境・エネルギー

**問題・疑惑**
- 犯罪・違反 / 政治資金問題 / 旧統一教会関係 / 学歴詐称 / 秘書問題

**発言・主張**
- 経済政策 / 防衛政策 / 社会政策 / 外交

## 7. 評価の更新手順

### 更新の基本フロー

```
1. Correctionファイルを作成
      ↓
2. merge_batches.py を実行
      ↓
3. generate_js_data.py を実行
      ↓
4. git commit → git push
      ↓
5. GitHub で PR 作成 → マージ
      ↓
6. GitHub Pages に自動デプロイ（数分後）
```

### Correctionファイルの書き方

`corrections/correction_034_議員名.json` のような名前で保存します。

#### 基本形（スコア・テキスト更新）

```json
[
  {
    "_action": "update",
    "_note": "P001 ○○○○: △△△△の実績によりスコア更新",
    "id": "P001",
    "axes": [4, 4, 3, 4, 3, 4, 4, 3],
    "total": 75,
    "rank": "B",
    "plus": "更新されたプラス評価テキスト",
    "minus": "更新されたマイナス評価テキスト",
    "_replace_evidence": false,
    "evidence": [
      {
        "cat": "実績",
        "sub": "政策実現",
        "summary": "一行要約（100字以内）",
        "detail": "具体的な詳細説明",
        "src": "Wikipedia",
        "url": "https://ja.wikipedia.org/wiki/○○",
        "rel": "政策実現力",
        "impact": "高",
        "date": "2025-10"
      }
    ]
  }
]
```

#### 特定フィールドのみ更新（例: minusテキストのみ）

```json
[
  {
    "_action": "update",
    "id": "P001",
    "minus": "修正後のマイナス評価テキスト"
  }
]
```

#### フラグの設定

```json
[
  {
    "_action": "update",
    "id": "P001",
    "flag_crime": true,
    "flag_caution": true,
    "axes": [4, 4, 3, 4, 2, 4, 4, 0],
    "total": 67,
    "rank": "C+"
  }
]
```

### _action の種類

| アクション | 意味 | 用途 |
|-----------|------|------|
| **update** | 指定フィールドのみ更新 | 通常の修正・追加 |
| **reset** | 評価を未評価状態に戻す | 情報が古くなった場合 |
| **add** | 新規議員を追加 | 議員増員時 |

### ローカルPCでの実行コマンド

```bash
# 1. マージ（correctionを適用）
python merge_batches.py

# 2. data.js を再生成
python generate_js_data.py

# 3. コミット
git add corrections/correction_034_*.json data.js output/politicians_base.json
git commit -m "update: ○○ スコア更新 — 根拠: △△△"

# 4. プッシュ
git push origin main
```

## 8. 見直しタイミングと優先順位

### 定期見直し（スケジュール）

| タイミング | 対象 | 内容 |
|-----------|------|------|
| **選挙直後** | 当選・落選した議員 | status（現職/元職）更新、再評価 |
| **内閣改造後** | 新大臣・副大臣 | 実績評価の追加、スコア見直し |
| **予算成立時** | 財政・経済担当議員 | 政策実現の根拠追加 |
| **報道発生時** | 不祥事が報じられた議員 | フラグ設定、証拠追加 |
| **Wikipedia更新時** | 情報不足議員 | Wikipedia基礎評価の再実行 |

### 優先見直し対象（高優先）

1. **犯罪フラグ議員（7名）**: 最新情報の確認（無罪・執行猶予などの更新）
2. **要注意フラグ議員（116名）**: 政治資金問題の解消・継続の確認
3. **情報不足議員（131名）**: 追加情報の収集・Wikipedia再取得
4. **高スコア議員（A-以上 12名）**: 根拠の質・信頼性の確認
5. **スコアと印象が乖離する議員**: ユーザーからのフィードバック対応

### Wikipedia基礎評価の再実行方法

131名の情報不足議員は、ローカルPCから以下を実行します（クラウドからはWikipedia APIへのアクセスが制限）:

```bash
python generate_wikipedia_batches.py
```

**注意**: レート制限を避けるため、DELAY=2.0秒の間隔が設定されています。429エラー（過多リクエスト）が発生した場合は自動リトライします。

## 9. 評価の客観性を守るための原則

### スコア変更のルール

| 変更の種類 | 必要なもの |
|-----------|-----------|
| スコアを上げる（+1軸） | 具体的な実績のEvidence（URL付き）が必要 |
| スコアを下げる（-1軸） | 具体的な問題のEvidence（URL付き）が必要 |
| テキスト（plus/minus）のみ変更 | 根拠不要だが、主観的表現は避ける |
| フラグの設定 | 信頼できる報道・公式記録が必要 |

### マイナス評価に含めてはいけないもの

以下の表現はマイナス評価に使ってはいけません：

| NG表現 | 理由 |
|--------|------|
| 「主張が強硬で軋轢を生む」 | プラスにもなりうる主観的評価 |
| 「発言が過激すぎる」 | 価値観による判断 |
| 「○○党所属のため信頼できない」 | 政治的バイアス |
| 「高齢であるため」 | 差別的評価 |
| 「知名度が低い」 | 評価軸と無関係 |

### マイナス評価に含めるべきもの

以下は具体的根拠があればマイナス評価に含めます：

- 政治資金の不記載・虚偽記載（政治倫理）
- 逮捕・起訴・有罪（政治倫理）
- 統一教会など反社会的組織との接点（公共性）
- 政策の明確な失敗（数値で示せるもの）
- 公約の不履行（根拠がある場合）

### 「主観」vs「客観」の判断基準

**客観的評価**: 誰が見ても同じ結論になる事実  
→ 「財務大臣として○年に△△を実現した」  

**主観的評価**: 見る人によって評価が変わるもの  
→ 「リーダーシップがある」「話が下手」

## 10. データフローと技術構成

### ファイル構成

```
portfolio/
├── data.js                    # フロントエンド向けデータ（自動生成）
├── politicians.html           # 議員一覧ページ
├── politician.html            # 議員詳細ページ
│
├── output/
│   ├── politicians_base.json  # 全816名のマスターデータ
│   ├── politicians_merged.json # マージ結果（中間ファイル）
│   └── wikipedia_cache.json   # Wikipediaキャッシュ
│
├── batches/
│   └── batch_*.json           # 15名ずつのWikipedia評価バッチ
│
├── corrections/
│   └── correction_*.json      # 手動修正・追加データ（番号順適用）
│
└── スクリプト
    ├── merge_batches.py       # バッチ+correctionをマージ
    ├── generate_js_data.py    # data.jsを生成

```

### データ更新の優先順位

correctionファイルはバッチより後に適用されるため、常にcorrectionが優先されます。

```
batches/batch_001.json  →  基礎データ
batches/batch_002.json  →  上書き
...
corrections/correction_001.json → バッチを上書き
corrections/correction_002.json → さらに上書き
...（番号順）
```

### GitHub Pages デプロイ

`main` ブランチへのプッシュ後、数分でサイトが自動更新されます。

## 付録A: よくある質問（FAQ）

**Q: 議員本人からクレームが来たらどうするか？**  
A: 証拠に基づいた評価であれば訂正不要。ただし事実誤認が確認された場合は速やかにcorrectionで修正する。

**Q: 複数の情報源が矛盾している場合は？**  
A: 一次情報（公式記録・裁判所文書）を優先。二次情報（報道）は複数メディアが一致している場合のみ採用。

**Q: 引退・死去した議員の扱いは？**  
A: status を「元職」に変更し、評価は保持（歴史的記録として残す）。

**Q: Wikipedia以外のソースが使えるか？**  
A: 官公庁の公式資料、衆議院・参議院の会議録、大手報道（NHK・朝日・読売・毎日・日経など）はすべて有効なソース。

**Q: スコアの上限（5点）はどんな議員か？**  
A: その軸で国内トップクラスの客観的実績がある議員。例: 財務大臣で歴史的な財政成果を残した場合は経済・財政軸5点。
